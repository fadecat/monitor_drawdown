# ETF 轮动 V2 每日邮件设计

## 目标

这封邮件服务 V2 实盘试运行，不做新策略研究。

核心目标是每天回答三个问题：

1. 今天的 V2 信号是否可用？
2. 当前应该持有什么？
3. 候选池分数和数据日期是否正常？

邮件每天发送。标的之间存在数据滞后差异：国内 ETF 净值通常是 T-1，跨境 ETF 可能是 T-2，指数和 ETF 净值也可能不是同一天更新。因此邮件必须先展示数据日期，再展示信号结论。

## 数据齐备口径

V2 runner 会先读取所有风险资产和防守资产的最新数据日期。

统一信号日定义为：

```text
signal_date = 所有标的 latest_date 的最小值
```

计算候选池时，所有标的都只使用 `signal_date` 及以前的数据，不允许混用较新的 A 股数据和较旧的跨境 ETF 数据。

如果所有标的都能回退到共同 `signal_date`，即使最新数据日期不完全一致，也可以生成 V2 信号。邮件会标记为：

```text
数据日期不一致，已回退统一信号日
```

只有在缺少标的、序列为空、无法形成共同 `signal_date` 时，才视为数据不可用，并沿用上一有效信号。

输出文件：

```text
.test_artifacts/etf_rotation_v2_strategy/data_status.json
```

关键字段：

| 字段 | 含义 |
|---|---|
| `signal_date` | 本次信号统一使用的数据日期 |
| `all_targets_aligned` | 所有标的最新数据日期是否完全一致 |
| `latest_dates_by_label` | 每个标的最新数据日期 |
| `lagging_labels` | 相对最新标的仍滞后的标的 |
| `status` | `ready` 或 `data_unavailable` |

## 邮件主题

主题行必须让手机通知栏也能看懂今天是否需要处理。

```text
【数据不可用】ETF轮动V2 | 沿用上一信号 | 信号日 2026-06-05
【持仓不变】ETF轮动V2 | 黄金ETF易方达 | 信号日 2026-06-05
【换仓通知】ETF轮动V2 | 黄金ETF易方达 → 纳指ETF国泰 | 信号日 2026-06-05
【切入防守】ETF轮动V2 | → 银华日利ETF | 信号日 2026-06-05
```

## 邮件正文

正文分三块。

第一块是今日决策：

- 今日结论
- 当前信号持仓
- 信号日
- 数据状态

第二块是数据状态：

- 统一信号日
- 滞后标的
- 每个标的最新数据日期

第三块是候选池评分：

- 排名
- 标的
- 数据日期
- `score_25`
- 年化收益
- `R²`
- `return_10d`
- 合格状态或拒绝原因

第四块是近 1 年策略收益率对比图。图表使用 PNG，不再使用 ASCII 文字曲线。策略线来自同一次运行中 V2 回测输出的 `daily_positions`；基准线使用沪深300ETF（510300）的复权净值 `adjUnitNav`，只作为展示基准，不进入 V2 候选池、排名、过滤或防守逻辑。

图表口径：

| 项目 | 口径 |
|---|---|
| 策略线 | V2 `strategy_nav` 在展示窗口首日归一化为 0% |
| 基准线 | 沪深300ETF（510300）`adjUnitNav` 在同一展示窗口首日归一化为 0% |
| 展示窗口 | 最近 365 个自然日内的共同交易日期 |
| 图表指标 | 策略近 1 年收益、基准近 1 年收益、超额收益、策略最大回撤 |

这个设计有两个约束：

1. 邮件信号、候选池评分和策略收益图使用同一份 V2 回测输出，避免“邮件信号来自今日抓取，图表来自另一套历史数据”的口径错配。
2. 图表只作为辅助展示。如果图表生成失败，邮件仍然照常生成，并回退显示已有的文字净值走势，避免辅助信息阻断每日持仓信号。

## 状态文件

邮件需要知道上一封有效信号持仓，才能判断今天是持仓不变还是换仓。

状态文件路径：

```text
data_state/etf_rotation_v2_email.json
```

只有 `status = ready` 并生成有效信号时才更新状态文件。

如果数据不可用，邮件照常发送，但不更新状态文件，避免用不完整数据覆盖上一有效信号。

## 运行归档

每次 GitHub Actions 运行成功后，除了发送邮件，还会把本次运行数据归档到：

```text
data_state/etf_rotation_v2_email/latest/
```

这个目录会随 Actions 提交到 GitHub。虽然目录名固定为 `latest`，每次运行会覆盖当前快照，但 Git 历史会保留每次运行后的版本，方便以后追查某一天的邮件和信号是怎么来的。

归档内容包括：

| 路径 | 内容 |
|---|---|
| `run_manifest.json` | 本次运行时间、邮件主题、信号日、数据状态 |
| `email_subject.txt` | 邮件主题 |
| `email_text.txt` | 纯文本邮件正文 |
| `email_preview.html` | HTML 邮件正文 |
| `state.json` | 发送后用于下次判断持仓变化的状态 |
| `source/` | 当次抓取并缓存的 ETF/指数序列数据 |
| `rotation/` | 当次 V2 runner 输出，包括 `data_status.json`、`candidate_metrics.json`、`portfolio_decision.json` |
| `backtest/` | 用同一份 `source/` 数据重跑的 V2 回测输出，包括 `daily_positions.csv`、`trades.csv`、`backtest_summary.md` |
| `benchmark/` | 当次沪深300ETF（510300）基准序列缓存 |
| `email_chart.png` | 邮件中展示的策略收益率 vs 沪深300ETF PNG 图 |

这样后续如果邮件信号、实盘操作或历史回测结果出现不一致，可以直接从 Git 历史中取回对应运行的完整输入和输出进行复现。

发送归档时优先复用邮件预览阶段已经生成的 `backtest/` 输出；只有缺少 `daily_positions.csv` 时才补跑回测。这样同一次 Actions 运行不会重复生成两份不同时间点的回测结果。基准序列和 PNG 图表也随同归档，方便以后复核邮件图表与当次数据是否一致。

## GitHub Actions

工作流：

```text
.github/workflows/etf_rotation_v2_email.yml
```

默认在 A 股收盘后运行：

```text
UTC 07:23 = Asia/Shanghai 15:23
```

发送成功后，如果 `data_state/etf_rotation_v2_email.json` 或 `data_state/etf_rotation_v2_email/` 发生变化，工作流会提交状态和归档数据，供下一次邮件判断持仓变化，也供后续追溯。

## 重要边界

这套邮件系统不改变 V2 策略规则。

它只做三件事：

1. 迁入 V2 稳定运行层。
2. 强制统一信号日，避免数据日期错配。
3. 把 V2 信号、候选池评分和数据状态用邮件发出来。

V3 / V4 / V4.1 不进入主线邮件系统。
